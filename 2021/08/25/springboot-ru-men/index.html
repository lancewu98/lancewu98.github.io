SpringBoot 入门

给属性赋值的方式

1.直接在属性上使用@Value(“属性值”)传值后自动注入（也可以写在setter上）

这个注解有个好处就是可以使用SpEL语法，但是需要一个个绑定，且不支持封装对象

2.在实体类上标注@ConfigurationProperties(prefix=”配置文件内的对象名”)后自动注入

这个注解的作用：将配置文件中配置的每一个属性的值，映射到这个组件中，将本类中的所有属性和配置文件中相关的配置进行绑定。注意：只有这个组件是容器中的组件，才能使用这个注解的功能

这种方式支持JSR303校验，在实体类上@Validated就可以在实体类的属性上加入一些注解进行数据校验（如@Email来校验是否是邮箱格式）

3.在实体类上标注@PropertySource(value = “自定义配置文件路径”)后自动注入

@ConfigurationProperties注解获取的是application.yaml或pplication.properties里的内容，这两个是被spring识别的。如果是非spring识别的配置文件，就需要用@PropertySource来读取。这种方式需要配合@Value来使用。

多环境配置及配置文件位置

springBoot会读取以下几个路径的application.yml：

  1. 当前文件的config文件夹内

  2. 当前文件下

  3. 类路径的config文件夹内

  4. 类路径下

优先级由上到下从高到低，同名配置项优先级高的会覆盖优先级低的。

被Spring识别的配置文件必须以application开头，以properties，yaml，yml为后缀，如：

application-dev.properties，application-test.properties..

这些配置文件里面可以设置的属性，其实就是自动配置加进来的配置类设定的属性，没有设定的话就取的自动配置类里面的默认值。至于哪些自动配置类是生效的，可以在application.properties里设置debug=true来查看控制台输出。

如果需要选择激活哪一套配置文件，可以使用spring.profiles.active来指定，如：

spring.profiles.active=dev

如果是yaml文件，可以把多套配置文件写在一起，配置文件之间用三条横杠—分开即可，然后每个配置文件内容可以用spring： profiles：XXX来命名。

这里看得出来有一个webjars的目录，可以在webjars的网站上，找到jquery的maven坐标引入一下。打开引入后的依赖包：

可以看到存在/META-INF/resources/webjars/的路径，和方法里的路径一致

addResourceHandlers方法继续往下看，有一行：

registration.addResourceLocations(this.resourceProperties.getStaticLocations());

其中的getStaticLocations()方法，一路点进去，最后会指向几个目录：

private static final String[] CLASSPATH_RESOURCE_LOCATIONS = &#123; "classpath:/META-INF/resources/","classpath:/resources/", "classpath:/static/", "classpath:/public/" &#125;;

所以这四个目录下的文件都可以被扫描，识别成静态资源。其中”classpath:/META-INF/resources/“就是我们上面看到的webjars的内容。另外三个目录的创建结构如下：

如果都在这三个路径下创建一个同名的js文件，内容各不相同，在浏览器直接访问localhost：8080/1.js就可以比较出这三个路径的优先级，从高到低分别是：resources>static（推荐）>public

以上是默认情况，如果在配置文件里配置了spring.mvc.static-path-pattern来指定静态资源加载路径，则默认路径下的资源不被扫描识别为静态资源，即addResourceHandlers方法的第一个if里的判断。

顺带一提，classpath目录下存在一个templates目录，这里面的所有页面只能通过controller来跳转，这个需要模板引擎的支持，即需要导入thymeleaf的依赖。

MVC配置

springboot支持使用自定义的mvc配置，只需要写一个@Configuration注解类，实现WebMvcConfigurer接口，且配置类上面不标注@EnableWebMvc注解即可，如下图是自定义一个MVC配置类，里面包含了一个自定义的视图解析器：

@Configuration
public class MyMvcConfig implements WebMvcConfigurer &#123;

    @Bean
    public ViewResolver myViewResolver()&#123;
        return new MyViewResolver();
    &#125;

    //自定义一个视图解析器
    //实现了视图解析器接口ViewResolver的类，就可以看做视图解析器
    public static class MyViewResolver implements ViewResolver&#123;
        @Override
        public View resolveViewName(String s, Locale locale) throws Exception &#123;
            return null;
        &#125;
    &#125;
&#125;

也可以如下写法，对url跳转对应的html视图：

@Configuration
public class MyMvcConfig implements WebMvcConfigurer &#123;

    //视图跳转
    @Override
    public void addViewControllers(ViewControllerRegistry registry) &#123;
        registry.addViewController("/hello").setViewName("test");
    &#125;
&#125;

注意：上面这个方法如果提示了 Cannot resolve MVC View ‘test’ ，说明缺少了thymeleaf的依赖，应在pom.xml内的依赖里补充如下内容：

 <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-thymeleaf</artifactId>
        </dependency>

回到开头，自定义MVC配置要求配置类上面不标注@EnableWebMvc注解，这是什么原因呢？可以点进这个注解看一下。

@Retention(RetentionPolicy.RUNTIME)
@Target(&#123;ElementType.TYPE&#125;)
@Documented
@Import(&#123;DelegatingWebMvcConfiguration.class&#125;)
public @interface EnableWebMvc &#123;
&#125;

可以看到这个注解引入了一个DelegatingWebMvcConfiguration类：

从这里可以看出这个类继承了WebMvcConfigurationSupport，它的作用是从容器中获取所有的WebMvcConfigurer。

回到MVC的自动配置类WebMvcAutoConfiguration：

可以看出上面有一个注解：@ConditionalOnMissingBean(WebMvcConfigurationSupport.class)

意思是容器里没有WebMvcConfigurationSupport这个类，MVC自动配置才生效，而DelegatingWebMvcConfiguration类继承了WebMvcConfigurationSupport，@EnableWebMvc又引入了DelegatingWebMvcConfiguration类。

这就是为什么自定义的MVC配置类，不允许加@EnableWebMvc注解。

异步任务

使用异步任务，只需要在SpringBoot的启动类上加上@EnableAsync，在需要异步处理的方法上标注@Async注解即可。
